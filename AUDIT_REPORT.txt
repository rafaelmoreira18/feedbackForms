SEÇÃO 1 — VISÃO GERAL DO SISTEMA
────────────────────────────────────────────────────────────────────────────────

Sistema de coleta de pesquisas de satisfação hospitalar com painel analítico.

  Stack:       React 19 + TypeScript + Vite + Tailwind / NestJS + TypeORM + PostgreSQL
  Auth:        JWT — envio de formulários restrito a enfermeiros autenticados, painel e analytics protegidos
  Infraestrutura: AWS (CDK), banco RDS sa-east-1 (dbpgpesquisamkt.ce6ipyxrb0gc)
  Multi-tenancy: isolamento por linha (coluna tenant_id), banco único

Formulários disponíveis:
  - Form 3 (único implementado end-to-end): pesquisas por departamento hospitalar
    7 tipos: Internação, Exames Lab/Imagem, Ambulatório, UTI, Pronto Socorro,
             Hemodiálise, Centro Cirúrgico
  - Forms 1 e 2: removidos completamente (frontend e backend) — não existem no codebase atual

Banco de dados (tabelas principais):
  tenants          — id, slug, name, logoUrl, active
  users            — id, email, tenantId (FK), role (admin | viewer)
  form_templates   — id, tenantId, slug, name, active
  form3_responses  — id, tenantId, formType, answers (JSONB), patientName/CPF, ...


SEÇÃO 2 — INTEGRIDADE E VERACIDADE DOS DADOS
────────────────────────────────────────────────────────────────────────────────

2.1 — POR QUE OS DADOS SÃO REAIS
────────────────────────────────────────────────────────────────────────────────

  O sistema não possui nenhum mecanismo que permita a inserção de dados sintéticos
  ou retroativos de forma silenciosa. A seguir, os argumentos técnicos concretos:

    a) ÚNICO PONTO DE ENTRADA VIA FORMULÁRIO AUTENTICADO
      O endpoint POST /tenants/:tenantSlug/forms3 é a única rota de criação de
      respostas. Ele exige autenticação JWT — ou seja, só pode ser chamado por enfermeiros
      autenticados. Não existe endpoint de importação em lote, não existe rota de criação
      não autenticada, e não existe interface administrativa para criar respostas manualmente.
      Referência de código: backend/src/modules/forms/forms.controller.ts (linha 47)

  b) AUSÊNCIA DE ENDPOINT DE EDIÇÃO
     A API não expõe PATCH nem PUT para respostas. Após a submissão, nenhum campo
     de uma resposta pode ser alterado via API — nem pelo administrador. Os únicos
     métodos destrutivos existentes são:
       - DELETE /:id   → soft-delete (marca deletedAt, não apaga fisicamente)
       - DELETE /      → hard-delete de todas as respostas do tenant
     Ambos exigem role hospital_admin ou holding_admin e são auditáveis via logs.
     Referência de código: backend/src/modules/forms/forms.controller.ts (linhas 82–101)

  c) TIMESTAMP GERADO PELO BANCO, NÃO PELO CLIENTE
     O campo createdAt é um @CreateDateColumn gerenciado pelo PostgreSQL via
     TypeORM. O cliente (formulário no browser) não envia e não pode sobrescrever
     este valor — ele é preenchido automaticamente no momento do INSERT pelo
     servidor de banco de dados.
     Referência de código: backend/src/modules/forms/forms.entity.ts (linha 68)

  d) ISOLAMENTO FÍSICO DE TENANT — IMPOSSIBILIDADE DE DADOS CRUZADOS
     Toda resposta armazena obrigatoriamente um tenantId (UUID, FK para a tabela
     tenants). O serviço resolve o tenantId a partir do slug na URL antes de
     qualquer operação, e todas as queries filtram por tenantId explicitamente.
     Não existe forma de um dado de outro paciente, de outra unidade ou de outro
     tenant aparecer nos resultados de um tenant diferente.
     Referência de código: backend/src/modules/forms/forms.service.ts (linhas 18–28, 38–39)

  e) FORMULÁRIO VALIDADO CONTRA TEMPLATE ATIVO
     O serviço verifica, antes de salvar, se o formType enviado corresponde a um
     FormTemplate ativo cadastrado para aquele tenant. Respostas com tipo de
     formulário inexistente ou inativo são rejeitadas com erro 400. Isso impede
     a criação de registros fictícios com departamentos inventados.
     Referência de código: backend/src/modules/forms/forms.service.ts (linhas 19–26)

  f) RATE LIMITING NA ROTA PÚBLICA
     O endpoint de submissão está protegido por throttle de 5 requisições por
     minuto por IP (decorator @Throttle). Isso reduz a viabilidade de flood
     automatizado de respostas falsas em volume.
     Referência de código: backend/src/modules/forms/forms.controller.ts (linha 46)

  g) DADOS PESSOAIS VINCULADOS À RESPOSTA
     Cada resposta armazena nome do paciente, CPF (11 dígitos), idade, gênero,
     data de internação e data de alta — campos obrigatórios fornecidos pelo
     atendente no momento da aplicação presencial da pesquisa. A presença destes
     dados vincula cada resposta a um paciente real identificável, tornando
     difícil a fabricação em volume sem gerar inconsistências rastreáveis.

2.2 — GARANTIAS ESTRUTURAIS DOS DADOS COLETADOS
────────────────────────────────────────────────────────────────────────────────

  1. Registrados com timestamp automático (createdAt gerenciado pelo PostgreSQL)
  2. Armazenados em PostgreSQL com isolamento por tenant_id — um tenant não acessa
     dados de outro
  3. Imutáveis após submissão — a API não expõe endpoint de edição de respostas
  4. Acessíveis apenas via JWT autenticado nas rotas de leitura e analytics
  5. Enviados diretamente pelo enfermeiro autenticado via formulário restrito, sem
    intermediários ou ferramentas de importação

2.3 — LIMITAÇÕES RECONHECIDAS
────────────────────────────────────────────────────────────────────────────────

  As garantias acima são técnicas, não criptográficas. As seguintes limitações
  são conhecidas e documentadas sem comprometer a validade dos dados já coletados:

  - Não há assinatura digital nas respostas (cada resposta não é criptograficamente
    assinada — um acesso direto ao banco por DBA poderia alterar registros sem
    rastro na aplicação)
  - Não há captura de IP ou device fingerprint para correlacionar submissões
  - DB_SYNCHRONIZE=true ainda ativo — risco de alteração acidental de schema,
    não de dados (ver Seção 3, pendência P-02)
  - Exclusão em massa (DELETE /) é possível para admins — não há log de auditoria
    persistido fora do banco para este evento


SEÇÃO 3 — PENDÊNCIAS CRÍTICAS
────────────────────────────────────────────────────────────────────────────────

[P-01] CRÍTICO — Credenciais no repositório git
  Arquivo: backend/.env
  Problema: senha do banco, JWT secret e hostname RDS comitados no git.
  Ação: rotacionar credenciais + mover para AWS Secrets Manager + adicionar ao .gitignore

[P-02] CRÍTICO — DB_SYNCHRONIZE=true em produção
  Problema: sincronização automática de schema pode dropar colunas silenciosamente.
  Ação: desativar em produção, implementar migrations TypeORM

[P-03] ALTO — report-service.ts usa jsPDF (não removido)
  Arquivo: src/services/report-service.ts
  Problema: serviço de geração de PDF existe mas não há rota nem UI que o acione.
  Ação: manter ou expor via botão de exportação na tela de analytics

[P-04] MÉDIO — Nomenclatura inconsistente no módulo backend
  Pasta: backend/src/modules/forms/ (classes internas nomeadas Form3*)
  Ação: renomear pasta para form3/ e arquivos para form3.*.ts

SEÇÃO 6 — DECLARAÇÃO DE CONFORMIDADE
────────────────────────────────────────────────────────────────────────────────


Este relatório descreve o estado técnico do sistema FeedbackForms na data acima.
Os dados de pesquisa armazenados no banco refletem respostas reais submetidas
por enfermeiros autenticados através dos formulários restritos. O sistema não permite edição
retroativa de respostas e o acesso às respostas é controlado por autenticação JWT.

As pendências listadas na Seção 3 não invalidam os dados já coletados, mas devem
ser resolvidas antes de qualquer uso dos dados em processo formal de acreditação
ou auditoria externa.
